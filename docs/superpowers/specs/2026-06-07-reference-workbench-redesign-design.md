# 竞品监控工具参考项目 UI 重设计规格

## 背景

当前项目已经完成过一轮“命令中心”方向的前端重构，界面具备完整功能，但整体表达仍然偏展示型：首页强调主视觉和大块状态卡，设置页偏悬浮大面板，和用户刚刚提供的参考项目 `C:\Users\Administrator\Desktop\小工具\坑位表` 相比，桌面工具感、紧凑度和执行导向还不够强。

本轮不是重新发明视觉语言，而是明确参考该项目的产品界面气质，对当前应用做一次“半重构”：

- 保留现有业务流程、字段、状态逻辑和 IPC 契约
- 首页与设置页一起统一重做
- 参考项目的布局骨架、信息密度、按钮体系、列表节奏和右侧执行摘要
- 不机械照抄颜色和业务内容，而是迁移“深色平面工具台”的产品感

## 已确认决策

- 参考强度：尽量整体参考 `坑位表` 的 UI 设计
- 改造范围：首页同步面板 + 设置页一起重做
- 改造方式：半重构
- 目标气质：紧凑、稳定、工具化、执行优先
- 保留约束：当前项目仍是 Amazon 竞品同步工具，不改同步流程与数据结构

## 设计目标

1. 首页进入后，用户应优先看到“这次要跑哪些分组、当前能不能跑、点击哪里开始跑”。
2. 首页从“命令中心展示卡”切换为“执行工作台”，减少叙述感，增强操作感。
3. 设置页从“两个风格化面板”切换为“左编辑、右运行参数”的维护台，密度和首页统一。
4. 所有高频操作入口必须稳定、靠近、可扫读，不再分散在多个视觉重心中。
5. 视觉上更接近参考项目的深色平面桌面工具，而不是保留当前偏发光、偏氛围化的卡片表达。

## 非目标

本轮不做以下内容：

- 不新增业务功能
- 不调整 `window.tracker` IPC 接口
- 不修改同步流程、日志落盘、截图落盘、Excel 输出逻辑
- 不引入新的导航层级或多页面结构
- 不扩展为整套应用壳改版，范围只覆盖首页与设置页

## 参考项目可迁移特征

从 `坑位表` 项目中，本轮明确迁移以下界面特征：

### 1. 页面骨架

- 顶部使用简洁状态条而非重型品牌头图
- 主内容采用“左主区 + 右侧摘要栏”的工作台布局
- 列表型任务区优先于英雄区
- 设置页延续双栏，但职责更硬朗清晰

### 2. 信息密度

- 使用连续列表承载分组状态
- 用小型摘要单元展示关键运行参数
- 压缩说明性文案，尽量一屏完成判断

### 3. 控件语言

- 主要按钮统一为紧凑矩形按钮，不再使用偏圆润、偏品牌化的大胶囊按钮
- 次级按钮保持统一边框和背景层级
- 删除/危险操作明确区分，但不戏剧化

### 4. 视觉语气

- 深色平面底色
- 更克制的层级对比
- 降低大面积半透明光感和装饰性高光
- 通过结构和密度建立专业感，而不是通过氛围感建立“高级”

## 首页重设计

## 首页目标

首页应当像一个执行工作台，而不是一个“命令中心海报”。它只回答四个问题：

1. 当前有哪些分组可跑
2. 这次准备跑哪些分组
3. 现在能不能开始同步
4. 同步后我去哪里看结果和异常

## 首页新结构

首页保留顶部状态条和 `同步面板 / 高级设置` 两个标签，但在 `同步面板` 内重排为两栏：

- 左侧主区：待同步分组列表
- 右侧侧栏：执行摘要 + 主动作 + 文件入口 + 异常留痕

不再保留当前的独立 Hero 区块作为首屏主视觉。

## 首页模块设计

### 1. 顶部状态条

保留品牌名、配置路径、站点、模式、分组总数等全局状态，但表达方式收敛为轻量顶栏：

- 品牌名弱化为应用名
- 站点 / 模式 / 分组数做成小型状态项
- 配置路径保留为辅助信息，不占主要视觉注意力

目标是为主工作区让位。

### 2. 待同步分组列表

当前右侧 `ExecutionRail` 中的分组卡片，改为首页左主区的连续列表主体。

每个分组行展示：

- 分组名
- 是否可执行
- ASIN 数量
- 简短预览信息
- 当前是否被选中

选中态通过背景和边框轻度强调，不再用厚卡片叠卡片的方式表达。

空分组继续显示，但弱化为“可见但不可执行”的行，不额外消耗大面积警告区域。

### 3. 执行摘要

右侧第一块改为执行摘要，内容聚焦本次同步：

- 已选分组数
- 可执行对象数量
- 当前站点
- 当前模式

这里替代当前首页大标题区域承担“执行前确认”的职责。

### 4. 主动作区

“开始同步任务”下沉到右侧摘要栏，成为唯一高权重动作。

同区域内保留：

- 开始同步任务
- 停止同步（如已有运行态）
- 进入设置页

不再单独为主按钮搭建一块高氛围感容器，而是让它成为工作台右栏中的主操作按钮。

### 5. 文件入口

当前“打开今日 Excel / 打开 data 目录 / 打开日志目录”统一为一组次级按钮，放入右栏下部。

要求：

- 文案直接
- 排列统一
- 不与主动作争抢视觉层级

### 6. 异常留痕

当前“异常留痕”区域继续保留，但收敛为小面板：

- 截图目录
- 价格调试目录
- 最近异常摘要

重点是让用户知道“去哪里查”，而不是让异常模块成为首页主角。

### 7. 活动日志

现有 `ActivityFeed` 继续保留，但视觉上要更接近执行日志面板，而不是叙述式内容区：

- 标题更直接，例如“运行轨迹”或“同步日志”
- 行高更紧凑
- 最新事件依旧在上方或末尾清晰可扫读
- 出错项与普通项有层级区分，但不过分染红整块区域

## 设置页重设计

## 设置页目标

设置页应是维护台，而不是首页风格的延长线。目标是让用户快速完成：

- 选分组
- 改分组名
- 维护 ASIN
- 配运行参数
- 保存设置

## 设置页新结构

延续现有双栏结构，但更贴近参考项目：

- 左栏：分组管理
- 右栏：运行参数

视觉上两栏都更紧凑、更平面，减少当前大量圆角、大面积柔光和漂浮感。

## 设置页模块设计

### 1. 分组管理区

将以下内容压回同一条编辑轨道：

- 当前分组选择
- 新建分组
- 删除当前分组
- 分组名称
- ASIN / Amazon 链接文本区
- 输出路径预览

输出路径预览改为底部弱提示，不再做成高亮提示卡。

### 2. 运行参数区

运行参数继续保留：

- Amazon 站点
- 有头/无头模式
- 美国邮编
- 首页等待时间
- 弹层等待时间
- 保存设置
- 恢复默认

但表达方式向参考项目靠拢：

- 表单项更规整
- 右栏更像参数面板
- “保存设置”是唯一主按钮
- “恢复默认”降级为次级操作

### 3. US / EU 条件切换

保留现有业务规则：

- `us` 显示邮编与等待时间项
- 欧洲站显示简化说明

但表现方式调整为参数面板内部切换，不额外制造大块提示噪音。

## 视觉系统调整

## 颜色

维持深色主题，但从当前“冷色 + 光感 + 半透明”转向“深色平面 + 结构层级”：

- 背景更接近中性深黑，而不是带明显氛围渐层的大底
- 面板层通过深浅和细边线区分
- 强调色继续可用，但范围收缩，只服务主动作和焦点态
- 异常色只用于危险态和问题提示

## 形状

- 卡片圆角统一收紧
- 按钮从偏胶囊改为更紧凑的矩形圆角
- 列表项和输入框形状统一

## 阴影与质感

- 减少大范围柔光阴影
- 减少玻璃感和过多 backdrop blur
- 更多依赖边框、层级和留白构建结构

## 排版

- 标题用更明确的产品界面层级
- 降低首页大标题的存在感
- 提升模块标题的执行语义
- 中文说明文本更短，尽量以任务为中心

## 组件映射建议

为降低改造成本，本轮优先在现有组件基础上重组，而不是全部推倒重写：

- `CommandCenterView`：整体改为参考项目式工作台布局
- `CommandHero`：弱化或拆解，其职责并入顶部状态条和右侧执行摘要
- `ExecutionRail`：拆成右栏摘要块与左侧分组列表的一部分
- `ActivityFeed`：保留但重新样式化
- `GroupEditorPanel`：重做为更紧凑的编辑台
- `CrawlerSettingsPanel`：重做为参数面板
- `SurfaceCard`：收紧圆角、阴影和背景策略，必要时只保留最小容器能力
- `index.css`：重写 token 和基础控件层，使整体质感向参考项目靠拢

## 交互规则

1. 首页主动作只有一个高权重入口。
2. 同步进行中，设置页仍按现有逻辑禁用相关输入和切页规则。
3. 分组列表的可执行/不可执行判断逻辑不变，只改变表现。
4. 保存失败、加载失败、恢复默认失败等错误提示继续保留，并纳入新视觉体系。
5. 所有新样式必须覆盖 hover、focus、disabled、busy 等状态。

## 测试与验收重点

实现后必须验证：

- `window.tracker` 缺失时的 preload 错误提示仍正确可见
- 同步面板在无可执行分组时主动作禁用且提示合理
- 同步进行中按钮、日志、右侧摘要状态正确
- 设置页保存成功 / 保存失败 / 恢复默认失败反馈仍正确
- `us / de / fr / it / es` 切换后的参数显隐逻辑不回归
- 窄屏下首页两栏与设置页两栏能自然折叠

## 验收标准

1. 首页首屏不再以大 Hero 为核心，而是明显呈现为工作台。
2. 首页视觉重心明确落在“待同步分组 + 执行摘要 + 开始同步任务”。
3. 设置页明显更紧凑，和首页属于同一套工具界面语言。
4. 用户能一眼看出这版更接近 `坑位表` 的工具气质，而不是当前版本的命令中心气质。
5. 现有功能、状态逻辑与测试覆盖不被破坏。
